# May 17: When the Training Data Doesn't Know Your Developer Tab

A Sunday conversation about AI hitting a wall on code that doesn't exist in the training data — and the creative workaround of asking it to build its own test harness.

<!-- more -->
<!-- enriched -->

## Developer Journal

### The problem: undocumented developer settings

A friend was working on a weekend project involving a Store app that has a developer settings tab not present in the retail build. The goal was to build something in that tab. The problem: the developer tab has poor error handling, throws unhandled runtime exceptions, and isn't documented anywhere accessible to an AI model.

The classic AI failure mode: the model knows how to write code for the public API of a well-documented framework. It doesn't know about the internal developer-only tab that crashed in three different ways before anything useful could even be attempted.

### The workaround: build the test harness first

Rather than trying to make the AI work directly against the broken surface, the instruction changed: create branches, build test harnesses, work incrementally. The AI started scaffolding test infrastructure that it could then use to characterize the exceptions before trying to fix them.

This is a useful inversion: instead of asking the AI to solve the problem directly, ask it to build the infrastructure that will let it understand the problem. For undocumented or edge-case code, "build me a way to see what's happening" is often a more productive prompt than "fix this."

The session was still going at the time of the conversation — iterating through `_copilot##` branches trying different approaches — but it was making progress. The unhandled exceptions in a developer-only tab are exactly the kind of thing that's not in any model's training data, so the model has to reason about it empirically rather than from prior knowledge.

### The parallel to JWST work

This maps directly to some of the FITS edge cases from May 12. The `RA=0, CRPIX1=0` WCS rejection wasn't in most AI training data as a known FITS idiom — it's a corner case that required looking at the raw spec rather than pattern-matching from examples. The magic byte validation similarly: knowing that `astropy.fits.open` produces confusing errors on non-FITS input is a detail that lives in Stack Overflow threads and project-specific knowledge, not in the general training distribution.

The lesson in both cases: AI works best on well-documented surfaces. When you're at the edge, the best move is often to build observability first rather than guess at solutions.
