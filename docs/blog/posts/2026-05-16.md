### The skills portability conversation

A conversation in the AI channel today surfaced an increasingly familiar frustration: building a sophisticated AI workflow in one tool and then discovering it doesn't transfer.

The specific case: Claude Code with a CLAUDE.md, custom skills in `.claude/skills/`, and carefully tuned instructions had developed into a system where you don't need to over-specify prompts. The skills encode working patterns. CLAUDE.md encodes project conventions. The agent carries the context forward session-to-session.

The equivalent in another tool (GitHub Copilot, Cursor) involves different file names, different instruction formats, different agent architectures. Conceptually similar — instructions.md, a knowledge base directory, skills files — but the specifics don't port. Even if the content could be migrated, the behaviors would differ because the underlying models and system prompt structures differ.

This is deliberate lock-in, and it's an interesting design choice. The tools benefit from making your workflow specifically theirs. The user benefits from the polish of a deeply integrated workflow. The cost shows up when you hit a limit (session ceiling, cost, coverage gap) and want to switch tools mid-task.

The partial mitigation: symlinks. If you're maintaining parallel setups in Claude and Copilot, `claude.md` and `instructions.md` can share content via symlink if the format is compatible. The skills won't translate automatically but the project context can stay synchronized.

For JWST, the setup is deep enough into Claude Code that full portability isn't practical. But noting the patterns that are tool-specific vs. genuinely portable is worth the exercise.
