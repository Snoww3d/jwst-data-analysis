# April 4: Security Foundations

Heavy security-infrastructure day — added the .NET exception hierarchy with centralized error middleware, security headers middleware on the gateway, gitleaks secret scanning in pre-commit and CI, plus the Claude Code GitHub workflow, a narrowed exception swallow in FITS save, and a full dependabot drop.

## Developer Journal

### Exception hierarchy and error middleware (#983)

The backend was catching `Exception` in way too many places — some of those catches rethrew unwrapped, some swallowed silently, some returned the raw `ex.Message` to clients. No clear policy about which exceptions are the caller's fault versus a bug versus a transient infrastructure problem.

Added a proper hierarchy:

- `JwstException` — base class
  - `ValidationException` — 400s, client input is wrong
  - `NotFoundException` — 404s, resource doesn't exist
  - `ConflictException` — 409s, state mismatch (e.g. editing a locked job)
  - `AuthorizationException` — 403s
  - `ExternalServiceException` — 502/503, MAST or upstream failed
  - `InternalException` — 500s, something we should fix

Centralized error middleware catches any `JwstException`, maps to the right status code, logs with appropriate severity, and returns a uniform error envelope (`{ code, message, traceId }`). Unhandled `Exception` still reaches a fallback that logs as Error and returns a generic 500 with a trace ID — no internal details leaked.

Narrowed the broad catches in the controllers that were obvious. The remaining ones are tracked as follow-ups in the exception infrastructure epic.

### MAST exception subtypes shipped as a 2-PR split

The MAST layer has its own failure modes — MAST API down, product not found in S3, auth credentials expired, archive temporarily offline for maintenance. These don't map cleanly to generic HTTP codes, so added `MastException` subtypes (`MastUnavailableException`, `MastProductNotFoundException`, etc.) and narrowed the catches in the MAST service layer to use them. PR 2 of 2 lands tomorrow.

### Security headers middleware (#954)

Added a small middleware that attaches the standard hardening headers to every response — `Strict-Transport-Security`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, and a conservative `Content-Security-Policy` for the API (no unsafe-eval, no inline scripts in responses that don't need them). Zero behavior change for clients that were already behaving, defense in depth for the ones that aren't.

### Gitleaks in pre-commit + CI (#955)

Secret scanning runs on every commit locally and on every push in CI. Configured the baseline to allow `docker/.env.example` (intentional placeholders only) and `*.test.ts` files where test-only tokens need to appear literally. The first run surfaced zero real secrets — confirmation that the existing habits around `.env` gitignoring and env-var loading have held up.

Issue #955 tracks tightening the gitleaks rules over time as the community rule set evolves.

### Claude Code GitHub workflow (#957, #958)

Added the Claude Code Action workflow to the repo. Two PRs because the first iteration (#957) didn't wire up the right secret scoping — the fix (#958) added the branch-level token restrictions. Now PRs can be reviewed by Claude with a comment trigger and the action has scoped, branch-bound credentials.

### Replace contextlib.suppress in save_fits_data (#981)

`save_fits_data` was using `contextlib.suppress(Exception)` around the actual file write — which meant a broken write silently did nothing and returned success. Replaced with an explicit `try/except` that catches `OSError` specifically, logs with context (filepath, size, free disk), and re-raises as a `JwstProcessingException`. No more silent FITS write failures.
